# Spec: Disaggregated Benchmarks Analysis

## 1. Executive Summary

This document outlines the product requirements for supporting the analysis of Disaggregated (Prefill/Decode separated) LLM serving benchmarks within Prism. It builds upon the core Prism vision by focusing on the specific needs of architecting and tuning disaggregated serving stacks, utilizing data from `llm-d-benchmark` and similar tools.

## 2. Benchmark Analysis & Context

### 2.1. Source Data Overview

We are analyzing benchmark reports generated by `llm-d-benchmark`. These reports provide detailed performance metrics for various configurations of the `llm-d` serving stack.

- **Key Variables:**
  - **Serving Architecture:** Aggregated (vLLM) vs. Disaggregated (Distaggregated/Split Prefill-Decode).
  - **Node Ratios:** For disaggregated setups, the ratio of Prefill (P) nodes to Decode (D) nodes (e.g., 1P:1D, 1P:2D, 2P:4D).
  - **Hardware:** Specific accelerator types (e.g., H100) and counts.
  - **Workload:** Input Sequence Length (ISL), Output Sequence Length (OSL), Request Rates.

### 2.2. Key Analysis Insights (from `analysis.ipynb`)

- **Trade-off Analysis:** The primary analytical goal is to understand the trade-off between **Time To First Token (TTFT)** and **Throughput**.
- **Pareto Frontiers:** Users need to identify the "Pareto frontier" of configurations—the set of configurations that offer the best possible throughput for a given latency constraint (or vice versa).
- **Bottleneck Identification:** Disaggregated stacks shift bottlenecks between prefill and decode phases depending on the workload (ISL vs OSL heavy).
- **Cost efficiency:** Different P:D ratios have different cost profiles.

## 3. User Stories

1.  **As a Platform Engineer**, I want to compare the performance of Aggregated vs. Disaggregated architectures for my specific workload (e.g., long context chat) so I can decide if the complexity of disaggregation is worth the performance gain.
2.  **As a Capacity Planner**, I want to find the optimal Ratio of Prefill to Decode nodes (P:D Ratio) that maximizes throughput while keeping TTFT under 200ms.
3.  **As an AI Architect**, I want to visualize the Pareto frontier of all tested configurations to ensure I am not choosing a sub-optimal deployment strategy.

## 4. Functional Requirements

### 4.1. Data Ingestion & Parsing

- **Support `llm-d-benchmark` Schema:** The platform must natively parse the JSON output from `llm-d-benchmark`.
  - _Schema File:_ `benchmark_report_v0.2_json_schema.json`
    - _Key Fields Mapping:_
      - `ttft`: `metrics.mean_ttft_ms`
      - `tpot`: `metrics.mean_tpot_ms` (or derived from throughput if missing)
      - `throughput`: `metrics.total_token_throughput` (tokens/sec) or `metrics.request_throughput` (req/sec)
      - `latency`: `metrics.mean_e2el_ms`
- **Serving Stack Metadata Extraction:**
  - **Folder Name Parsing:** The application must parse the parent directory name to extract configuration:
    - **Disaggregated:** `setup_modelservice_NA_NA_<P_Node>_<P_TP>_<D_Node>_<D_TP>` (e.g., `1_4_3_4` -> 1 Prefill Node @ TP4, 3 Decode Nodes @ TP4).
    - **Aggregated:** `setup_standalone_<Node>_<TP>_NA_NA_NA_NA` (e.g., `1_2` -> 1 Node @ TP2).
  - **P:D Ratio Calculation:**
    - If Disaggregated: Ratio = `P_Node` : `D_Node`. Use this string (e.g., "1:3") as a filterable dimension.
    - If Aggregated: Ratio = "N/A" or "Aggregated".

### 4.2. Filtering & Selection

- **Architecture Toggle:** A high-level filter to toggle between "Aggregated" and "Disaggregated" benchmarks.
- **P:D Ratio Filter:** A multi-select filter for specific ratios (e.g., "1:1", "1:2", "1:4"). This should be conditional or clearly grouped under Disaggregated options.
- **Scenario Grouping:** Ability to group benchmarks by "Scenario" (e.g., same Model + Hardware + Workload, varying only the P:D ratio).

### 4.3. Visualization

#### 4.3.1. Trade-off Scatter Plot (The "Pareto" Chart)

- **X-Axis:** Throughput (Tokens/Sec or Requests/Sec).
- **Y-Axis:** Latency (Mean TTFT or P99 TTFT).
- **Grouping/Color:** Points should be colored by P:D Ratio or Architecture.
- **Frontier Line:** Optionally draw a line connecting the Pareto-optimal points to highlight the efficiency frontier.

#### 4.3.2. Ratio Comparison Bar Chart

- For a fixed workload (e.g., ISL=200, OSL=200):
  - Side-by-side bars comparing Throughput and TTFT for different P:D ratios (e.g., Aggregated vs 1:1 vs 1:2).

## 5. User Interface Design Notes

- **Filter Panel:** Ensure "P/D Disaggregation" and "P/D Node Ratio" are prominent when the serving stack supports it.
- **Tooltips:** Hovering over a data point in the scatter plot should show the exact P:D config and the specific throughput/latency values.

## 6. Success Metrics

- Users can successfully import an `llm-d` benchmark report.
- Users can identify the optimal P:D ratio for a provided sample workload using the dashboard charts.
